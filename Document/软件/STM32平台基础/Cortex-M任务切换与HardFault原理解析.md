# Cortex-M 任务切换与 HardFault 原理解析

---

## 一、基础概念：寄存器与栈

### 1.1 Cortex-M 寄存器一览

| 寄存器 | 用途 |
| :--- | :--- |
| R0-R3 | 函数参数/返回值，调用者保存（Caller-saved） |
| R4-R11 | 局部变量，被调用者保存（Callee-saved） |
| R12 | 临时寄存器（IP，Intra-Procedure call） |
| R13 (SP) | 栈指针（MSP / PSP） |
| R14 (LR) | 链接寄存器（函数返回地址） |
| R15 (PC) | 程序计数器 |
| xPSR | 状态寄存器 |

通用寄存器 R0-R12 共 13 个，加上 SP、LR、PC、xPSR，就是 CPU 的**全部可编程状态**。

### 1.2 寄存器与内存的关系

- 寄存器是 **CPU 内部的硬件资源**，固定 13 个通用寄存器，不在内存中，访问速度极快（1 个时钟周期）
- 栈是 **RAM 中的一块普通内存区域**，通过 SP（栈指针）追踪当前位置
- 编译器在编译阶段决定哪些变量放寄存器、哪些放栈内存，寄存器放不下的变量直接放在栈内存中

```
任务的完整状态 = 寄存器快照 + 栈内存
                  ↑               ↑
              需要保存/恢复    一直在内存里，不会丢
              (R0-R12,LR,      (SP 指对了就能找到)
               PC,xPSR)
```

### 1.3 双栈指针机制

Cortex-M 有两个栈指针：

| 栈指针 | 用途 |
| :--- | :--- |
| MSP (Main Stack Pointer) | 主栈指针，用于异常/中断处理 |
| PSP (Process Stack Pointer) | 进程栈指针，用于任务运行 |

RTOS 中，每个任务在线程模式下使用 **PSP**，进入异常后自动切换到 **MSP**。通过异常返回时 LR 中的 **EXC_RETURN** 值决定返回后使用哪个栈指针。

## 二、入栈与出栈

ARM Cortex-M 的栈是**从高地址向低地址增长（Full Descending）**。

### 2.1 入栈（PUSH）

**入栈 = SP 下移（向低地址）+ 将数据写入 SP 指向的内存**

```
执行 PUSH {R4}：

① SP = SP - 4          （指针往下移 4 字节）
② 把 R4 的值写到 SP 指向的地址

执行前：                执行后：
          SP → [  空  ]           [  空  ]
               [  空  ]    SP → [ R4的值 ]
               [  旧数据 ]        [  旧数据 ]
```

数据来源不限于寄存器，也可以来自 RAM，但在任务切换场景下基本都是保存 CPU 寄存器的值。

### 2.2 出栈（POP）

**出栈 = 从 SP 指向的内存读出数据 + SP 上移（向高地址）**

```
执行 POP {R4}：

① 从 SP 指向的地址读出值，放入 R4
② SP = SP + 4          （指针往上移 4 字节）

执行前：                执行后：
                                  SP → [  旧值  ]（残留在内存，逻辑上无效）
          SP → [ R4的值 ]              [  旧数据 ]
               [  旧数据 ]
```

出栈后的数据去向由指令决定：`POP {R4}` 给 R4，`POP {PC}` 给 PC，等等。

### 2.3 入栈出栈对比

| | 数据方向 | SP 方向 |
| :--- | :--- | :--- |
| 入栈（PUSH） | 数据 → 栈内存 | SP 下移（低地址） |
| 出栈（POP） | 栈内存 → 目标位置 | SP 上移（高地址） |

### 2.4 注意区分：压栈 vs 分配栈空间

| 操作 | SP 变化 | 是否写入数据 | 名称 |
| :--- | :--- | :--- | :--- |
| `PUSH {R4}` | SP 下移 4 字节 | 是 | **压栈/入栈** |
| `SUB SP, #40` | SP 下移 40 字节 | 否，只预留空间 | **分配栈空间** |

压栈一定伴随数据写入，单纯移动 SP 只是分配空间，不算压栈。

## 三、RTOS 任务切换：上下文保存与恢复

RTOS 任务切换通常通过 **PendSV** 异常实现，上下文保存分为**硬件自动保存**和**软件手动保存**两个阶段。

### 3.1 硬件自动压栈（PendSV 触发瞬间）

硬件自动将以下 **8 个寄存器**从 CPU 压入当前任务栈（PSP）：

| 栈偏移 | 寄存器 |
| :--- | :--- |
| +0x1C | xPSR |
| +0x18 | PC（返回地址） |
| +0x14 | LR（R14） |
| +0x10 | R12 |
| +0x0C | R3 |
| +0x08 | R2 |
| +0x04 | R1 |
| +0x00 | R0 |

硬件**只自动保存 R0-R3、R12、LR、PC、xPSR 这 8 个**，不是 R0-R12 全部。

### 3.2 软件手动压栈（PendSV Handler 中）

剩余的 **R4-R11** 共 8 个寄存器，由 RTOS 在 PendSV Handler 中手动压栈：

```asm
MRS    R0, PSP          ; 获取当前任务的 PSP
STMDB  R0!, {R4-R11}    ; 手动保存 R4-R11 到任务栈
```

### 3.3 完整栈帧布局

```
高地址
┌──────────┐
│   xPSR   │  ← 硬件最先压入（最高地址）
│   PC     │
│   LR     │
│   R12    │
│   R3     │
│   R2     │
│   R1     │
│   R0     │  ← 硬件自动压栈完成后 PSP 指向这里
├──────────┤
│   R11    │  ← 软件开始手动压栈
│   R10    │
│   R9     │
│   R8     │
│   R7     │
│   R6     │
│   R5     │
│   R4     │  ← 全部压栈完成后 SP 指向这里（栈顶）
└──────────┘
低地址
```

### 3.4 完整切换流程

1. **PendSV 触发** → 硬件自动把 R0-R3, R12, LR, PC, xPSR 压入当前任务栈（PSP）
2. **PendSV Handler** → 软件手动把 R4-R11 压入当前任务栈
3. **保存 SP** → 将当前 SP 值存入当前任务的 TCB（任务控制块）
4. **切换 TCB** → 调度器选出下一个任务，从其 TCB 取出保存的 SP 值
5. **恢复 R4-R11** → 软件手动从新任务栈中弹出 R4-R11
6. **更新 PSP** → `MSR PSP, R0`
7. **异常返回** → 硬件自动从栈中恢复 R0-R3, R12, LR, PC, xPSR，CPU 从恢复的 PC 地址继续执行

## 四、HardFault 异常

### 4.1 压栈机制

HardFault 触发时，硬件同样**自动压入 8 个寄存器**（R0-R3、R12、LR、PC、xPSR），布局与任务切换时硬件压栈的部分完全一致。

**没有软件压栈 R4-R11 这一步**，因为 HardFault 通常不需要恢复执行，只需要保存崩溃现场供调试。不过 R4-R11 在进入 HardFault Handler 时还在寄存器里，用调试器可以直接查看。

### 4.2 判断栈帧位置

关键问题：栈帧被压到了 MSP 还是 PSP？通过 LR（EXC_RETURN）的 bit[2] 判断：

| EXC_RETURN 值 | bit[2] | 含义 |
| :--- | :--- | :--- |
| 0xFFFFFFF1 | 0 | 返回 Handler 模式，使用 MSP |
| 0xFFFFFFF9 | 0 | 返回 Thread 模式，使用 MSP |
| 0xFFFFFFFD | 1 | 返回 Thread 模式，使用 PSP |

### 4.3 定位崩溃现场

汇编部分：

```asm
HardFault_Handler:
    TST    LR, #4           ; 测试 EXC_RETURN 的 bit[2]
    ITE    EQ
    MRSEQ  R0, MSP          ; bit[2]=0，栈帧在 MSP
    MRSNE  R0, PSP          ; bit[2]=1，栈帧在 PSP
    BL     HardFault_Handler_C
```

C 函数部分：

```c
void HardFault_Handler_C(uint32_t *stack_frame) {
    uint32_t r0   = stack_frame[0];
    uint32_t r1   = stack_frame[1];
    uint32_t r2   = stack_frame[2];
    uint32_t r3   = stack_frame[3];
    uint32_t r12  = stack_frame[4];
    uint32_t lr   = stack_frame[5];  // 异常前的 LR
    uint32_t pc   = stack_frame[6];  // 崩溃的指令地址
    uint32_t xpsr = stack_frame[7];
    while(1);
}
```

其中 `stack_frame[6]`（PC）是最关键的信息——它指向导致 HardFault 的那条指令（或附近）。

### 4.4 常见 HardFault 原因

- 访问非法地址（未映射的内存区域）
- 未对齐的内存访问
- 在异常中执行了 SVC 指令
- 除零（需在 SCB->CCR 中使能 DIV_0_TRP）
- 栈溢出（SP 跑飞到非法区域）

## 五、任务切换与 HardFault 对比

| | RTOS 任务切换 | HardFault |
| :--- | :--- | :--- |
| 触发方式 | 软件主动触发 PendSV | 硬件错误被动触发 |
| 硬件压栈 | R0-R3, R12, LR, PC, xPSR（8 个） | R0-R3, R12, LR, PC, xPSR（8 个） |
| 软件压栈 | R4-R11（8 个） | 无 |
| 使用的栈 | PSP（任务栈） | 取决于异常前模式（PSP 或 MSP） |
| 目的 | 保存/恢复上下文，任务继续执行 | 保存崩溃现场，供调试定位 |
| 返回 | 正常恢复并继续执行 | 通常不返回（死循环或复位） |

**本质上是同一套硬件压栈机制，任务切换用来保存和恢复任务状态，HardFault 用来保存崩溃现场供分析。**

## 六、FPU 扩展（Cortex-M4/M7）

如果使用了浮点单元（FPU），上下文还需要额外保存：

- 硬件自动压栈：S0-S15 + FPSCR
- 软件手动压栈：S16-S31

开启 FPU 后每个任务的栈空间需要更大。FreeRTOS 中通过 `configENABLE_FPU` 配置。
