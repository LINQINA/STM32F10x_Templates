## 1. 设计目标 (Design Goals)

本驱动旨在提供独立看门狗（IWDG）的初始化与喂狗功能，防止程序因死循环、跑飞等异常情况导致系统无法恢复：

1. **防卡死保护**：程序异常时，看门狗超时自动复位 MCU。
2. **独立时钟**：使用 LSI（40kHz），不依赖主时钟，即使主时钟故障也能触发复位。
3. **RTOS 友好**：喂狗操作分布在空闲钩子和周期任务中，保证多任务场景下的可靠性。

## 2. 硬件原理 (Hardware Principle)

### 2.1 独立看门狗（IWDG）vs 窗口看门狗（WWDG）

| 特性 | IWDG（独立看门狗） | WWDG（窗口看门狗） |
| :--- | :--- | :--- |
| **时钟源** | LSI（~40kHz），独立于主时钟 | APB1（36MHz），依赖主时钟 |
| **计数器** | 12 位递减计数器 | 7 位递减计数器 |
| **喂狗限制** | 只有上限（超时前喂狗即可） | 有上限和下限（必须在窗口区间内喂狗） |
| **下限值** | 无 | 固定 0x3F，不可修改 |
| **可检测异常** | 程序卡死、跑飞 | 程序卡死 + 程序执行过快（逻辑异常） |
| **启动后能否关闭** | 不能，一旦启动只能喂狗 | 不能 |
| **本项目是否使用** | ✅ 使用 | ❌ 未使用 |

### 2.2 IWDG 工作原理

```
         ┌──────────────┐
  LSI ──►│  预分频器     │──►  12位递减计数器
 40kHz   │  (÷4~÷256)   │     [Reload → 0]
         └──────────────┘          │
                                   ▼
                            计数器 = 0 ?
                            ┌───┴───┐
                            │  是   │ ──► 触发系统复位
                            │  否   │ ──► 继续递减
                            └───────┘
         喂狗操作：将计数器重新装载为 Reload 值
```

**超时时间计算公式**：

$$T_{out} = \frac{Prescaler \times Reload}{f_{LSI}}$$

其中：
- $f_{LSI}$ = 40kHz（典型值，实际范围 30~60kHz）
- $Prescaler$ = 预分频系数
- $Reload$ = 重装载值（12 位，最大 4095）

### 2.3 本项目实际配置

| 参数 | 值 | 说明 |
| :--- | :--- | :--- |
| **预分频系数** | 256 | `IWDG_PRESCALER_256` |
| **重装载值** | 2500 | 12 位范围内（最大 4095） |
| **LSI 频率** | 40kHz（典型值） | |
| **超时时间** | **16s** | 256 × 2500 / 40000 = 16s |

> 16 秒的超时时间较为宽裕，适合在 RTOS 多任务环境下使用。即使某个高优先级任务长时间占用 CPU，空闲任务和传感器任务仍有足够的时间窗口完成喂狗。

### 2.4 WWDG 工作原理（本项目未使用，仅作知识记录）

```
计数器值:  0x7F -----> W -----> 0x3F -----> 0x3F以下
           |          |         |            |
           初始值      窗口上限   固定下限      复位
           |          |         |
           这段喂狗→复位  这段喂狗→OK  没喂狗→复位
           (太早)      (合法窗口)   (太晚)
```

- **窗口上限值 W**：可配置，范围 0x40~0x7F
- **窗口下限值**：固定 0x3F，硬件决定，不可修改
- **合法喂狗区间**：W ≥ 计数器当前值 > 0x3F
- **应用场景**：对时序要求严格的系统，既能检测程序"太慢"（卡死），也能检测程序"太快"（逻辑异常）

## 3. 软件实现细节 (Implementation Details)

### 3.1 对外 API

| 函数名 | 说明 |
| :--- | :--- |
| `vWatchdogInit()` | 初始化 IWDG 并启动看门狗 |
| `vWatchdogReload()` | 喂狗（刷新计数器） |

### 3.2 初始化流程

```c
IWDG_HandleTypeDef g_iwdg_handle;

void vWatchdogInit(void)
{
    g_iwdg_handle.Instance = IWDG;
    g_iwdg_handle.Init.Prescaler = IWDG_PRESCALER_256;  /* 分频系数 256 */
    g_iwdg_handle.Init.Reload = 2500;                   /* 重装载值 2500 */
    HAL_IWDG_Init(&g_iwdg_handle);                      /* 初始化并启动 */
}
```

`HAL_IWDG_Init` 内部会完成以下操作：
1. 写入 `IWDG_KEY_ENABLE`（0xCCCC）启动看门狗
2. 写入 `IWDG_KEY_WRITE_ACCESS_ENABLE`（0x5555）允许写入预分频和重装载寄存器
3. 配置预分频系数和重装载值
4. 等待寄存器更新完成
5. 写入 `IWDG_KEY_RELOAD`（0xAAAA）刷新计数器

> **注意**：一旦 IWDG 启动，就无法通过软件关闭，只能持续喂狗。调试时需要在调试器配置中冻结 IWDG，否则断点暂停会导致看门狗复位。

### 3.3 喂狗实现

```c
void vWatchdogReload(void)
{
    if(g_iwdg_handle.Instance != NULL)
        HAL_IWDG_Refresh(&g_iwdg_handle);
}
```

增加了 `Instance != NULL` 的判断，确保在未初始化时调用不会出错。

### 3.4 喂狗策略：双保险机制

本项目采用**两个位置同时喂狗**的策略：

```
┌─────────────────────────────────────────────────────┐
│                    喂狗位置 1                         │
│          vApplicationIdleHook() (空闲钩子)            │
│          - CPU 空闲时自动调用                         │
│          - 保证系统正常运行时持续喂狗                   │
├─────────────────────────────────────────────────────┤
│                    喂狗位置 2                         │
│          vTaskSensor() (传感器任务, 5ms 周期)         │
│          - 周期性任务中主动喂狗                        │
│          - 即使空闲任务被长时间抢占，也能保证喂狗       │
└─────────────────────────────────────────────────────┘
```

**为什么用双保险？**

| 场景 | 仅靠空闲钩子 | 仅靠传感器任务 | 双保险 |
| :--- | :--- | :--- | :--- |
| 高优先级任务长时间占用 CPU | ❌ 空闲任务得不到执行 | ✅ 传感器任务有更高优先级 | ✅ 安全 |
| 传感器任务异常阻塞 | ✅ 空闲任务仍能执行 | ❌ 喂狗停止 | ✅ 安全 |
| 全部任务死锁 | ❌ 无法喂狗，触发复位 | ❌ 无法喂狗，触发复位 | ✅ 正确触发复位 |

### 3.5 当前启用状态

```c
// taskSystem.c → vUserSystemInit()

```

在开发调试阶段，看门狗初始化被注释掉。正式发布时取消注释即可启用。

## 4. 常见问题与知识点 (Q&A)

### Q1: LSI 频率不准（30~60kHz），超时时间会偏差多少？

**A**：以本项目配置为例：
- LSI = 30kHz 时：256 × 2500 / 30000 ≈ **21.3s**
- LSI = 40kHz 时：256 × 2500 / 40000 = **16.0s**
- LSI = 60kHz 时：256 × 2500 / 60000 ≈ **10.7s**

最坏情况下约 10.7 秒，仍然远大于 5ms 的传感器任务周期，不会误触发。看门狗不需要高精度，只要保证"程序没死就不复位"即可。

### Q2: 调试时断点暂停会触发看门狗复位怎么办？

**A**：在 Keil 调试配置中添加以下初始化脚本，进入调试模式时冻结 IWDG：

```ini
// STM32F1 Loader.ini
// DBGMCU_CR: DBG_IWDG_STOP = 1
*((uint32_t *)0xE0042004) |= ((uint32_t)1 << 8)
```

也可以在代码中通过 HAL 宏配置：
```c
__HAL_DBGMCU_FREEZE_IWDG();
```

### Q3: 为什么不使用窗口看门狗（WWDG）？

**A**：
1. 本项目运行在 FreeRTOS 多任务环境下，各任务执行时间受调度器影响，不适合对喂狗时间做严格的"下限"约束。
2. 独立看门狗的 LSI 时钟独立于主时钟，即使主振荡器故障也能工作，可靠性更高。
3. 16 秒的超时窗口足以覆盖所有正常运行场景。

### Q4: 如果想修改超时时间怎么算？

**A**：使用公式 $T_{out} = \frac{Prescaler \times Reload}{f_{LSI}}$，常用配置参考表：

| 预分频系数 | Reload | 超时时间 (LSI=40kHz) |
| :--- | :--- | :--- |
| 4 | 4095 | 409.5ms |
| 16 | 4095 | 1.638s |
| 64 | 4095 | 6.55s |
| 256 | 625 | 4s |
| 256 | 1250 | 8s |
| 256 | 2500 | **16s（当前配置）** |
| 256 | 4095 | 26.2s（最大值） |

## 5. 移植与扩展 (Porting Guide)

### 5.1 修改超时时间

修改 `vWatchdogInit()` 中的预分频和重装载值：

```c
g_iwdg_handle.Init.Prescaler = IWDG_PRESCALER_64;   // 改为 64 分频
g_iwdg_handle.Init.Reload = 2500;                    // 超时 = 64×2500/40000 = 4s
```

### 5.2 正式发布时启用看门狗

取消 `taskSystem.c` 中 `vUserSystemInit()` 里的注释：

```c
vWatchdogInit();   // 取消注释，启用独立看门狗
```

### 5.3 增加看门狗复位原因检测

可在系统启动时检测上次复位是否由看门狗触发，方便排查问题：

```c
if(__HAL_RCC_GET_FLAG(RCC_FLAG_IWDGRST) != RESET)
{
    printf("系统由独立看门狗复位\r\n");
    __HAL_RCC_CLEAR_RESET_FLAGS();
}
```

### 5.4 移植到 GD32 平台

GD32 的独立看门狗称为 FWDGT（Free Watchdog Timer），API 不同但原理一致：

| STM32 HAL | GD32 标准外设库 |
| :--- | :--- |
| `HAL_IWDG_Init()` | `fwdgt_config()` + `fwdgt_enable()` |
| `HAL_IWDG_Refresh()` | `fwdgt_counter_reload()` |
| LSI 40kHz | IRC40K 40kHz |

```c
// GD32 版本
void vWatchdogInit(void)
{
    rcu_osci_on(RCU_IRC40K);
    while(SUCCESS != rcu_osci_stab_wait(RCU_IRC40K));
    fwdgt_config(2500, FWDGT_PSC_DIV256);
    fwdgt_enable();
}

void vWatchdogReload(void)
{
    fwdgt_counter_reload();
}
```
