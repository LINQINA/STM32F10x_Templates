## 2. SPI 基础原理 (SPI Fundamentals)

### 2.1 什么是 SPI？

SPI (Serial Peripheral Interface) 是由 Motorola 提出的 **四线式全双工同步串行总线**，广泛用于板级高速通信。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         SPI 总线拓扑结构                                      │
└─────────────────────────────────────────────────────────────────────────────┘

                    ┌──────────┐           ┌──────────┐
                    │   主机    │   MOSI    │   从机1   │
                    │  (MCU)   │──────────►│  (Flash)  │
                    │          │   MISO    │           │
                    │          │◄──────────│           │
                    │          │   SCLK    │           │
                    │          │──────────►│           │
                    │          │   CS0     │           │
                    │          │──────────►│           │
                    │          │           └──────────┘
                    │          │
                    │          │           ┌──────────┐
                    │          │   MOSI    │   从机2   │
                    │          │──────────►│  (LCD)    │
                    │          │   MISO    │           │
                    │          │◄──────────│           │
                    │          │   SCLK    │           │
                    │          │──────────►│           │
                    │          │   CS1     │           │
                    │          │──────────►│           │
                    │          │           └──────────┘
                    └──────────┘

每个从机需要独立的 CS 线，MOSI/MISO/SCLK 可共享。
```

### 2.2 四根信号线

| 信号线 | 全称 | 方向 | 功能 |
| :--- | :--- | :--- | :--- |
| **MOSI** | Master Out Slave In | 主机 → 从机 | 主机发送数据 |
| **MISO** | Master In Slave Out | 从机 → 主机 | 从机发送数据 |
| **SCLK** | Serial Clock | 主机 → 从机 | 时钟信号，由主机产生 |
| **CS (NSS)** | Chip Select | 主机 → 从机 | 片选，**低电平有效**，选中目标从机 |

```
SPI 全双工通信示意:

    ┌──────┐                          ┌──────┐
    │      │── MOSI ────────────────►│      │
    │ 主机  │◄─ MISO ─────────────────│ 从机  │
    │ MCU  │── SCLK ────────────────►│ Flash │
    │      │── CS ──────────────────►│      │
    └──────┘                          └──────┘
         │                                │
         │◄── 数据同时双向传输 ──►│
         │     (全双工)                   │

MOSI 和 MISO 是独立的两根线，主机发送的同时可以接收。
```

### 2.3 片选机制 — 硬件寻址

SPI 的寻址方式非常直接：**主机拉低 CS 线 = 选中从机**，无需软件地址，零开销。

```
主机选中从机1 (拉低 CS0):

    CS0: ──┐                                         ┌──
           └─────────────────────────────────────────┘
           ↑                                         ↑
        选中从机1                                  释放从机1
        
    CS1: ────────────────────────────────────────────────
                        (保持高电平，从机2 未被选中)

    SCLK:    ┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐  ┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐
          ───┘└┘└┘└┘└┘└┘└┘└┘└──┘└┘└┘└┘└┘└┘└┘└┘└──
                 字节1                字节2

    MOSI:  ──[  D7 D6 D5 D4 D3 D2 D1 D0 ]──[........]──

从机1: CS0 被拉低 → 开始工作，响应时钟，收发数据
从机2: CS1 保持高 → 忽略总线上所有信号
```

### 2.4 SPI 四种工作模式

SPI 有 4 种模式，由 **CPOL** (时钟极性) 和 **CPHA** (时钟相位) 两个参数组合决定：

| 模式 | CPOL | CPHA | 时钟空闲电平 | 数据采样边沿 | 数据输出边沿 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Mode 0** | 0 | 0 | 低电平 | 上升沿采样 | 下降沿输出 |
| **Mode 1** | 0 | 1 | 低电平 | 下降沿采样 | 上升沿输出 |
| **Mode 2** | 1 | 0 | 高电平 | 下降沿采样 | 上升沿输出 |
| **Mode 3** | 1 | 1 | 高电平 | 上升沿采样 | 下降沿输出 |

```
Mode 0 (CPOL=0, CPHA=0) — 最常用:

SCLK:     ┌──┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐
      ────┘  └──┘  └──┘  └──┘  └──┘  └──┘  └──┘  └──┘  └────
          ↑     ↑     ↑     ↑     ↑     ↑     ↑     ↑
        采样   采样   采样   采样   采样   采样   采样   采样
        (上升沿)

MOSI: ──[D7]──[D6]──[D5]──[D4]──[D3]──[D2]──[D1]──[D0]──
         ↑                                              ↑
       MSB 先发                                       LSB


Mode 3 (CPOL=1, CPHA=1) — 也常用:

SCLK: ────┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐
          └──┘  └──┘  └──┘  └──┘  └──┘  └──┘  └──┘  └────
             ↑     ↑     ↑     ↑     ↑     ↑     ↑     ↑
           采样   采样   采样   采样   采样   采样   采样   采样
           (上升沿)

MOSI: ────[D7]──[D6]──[D5]──[D4]──[D3]──[D2]──[D1]──[D0]──
```

> **模式选择原则**：查看从机芯片的数据手册，芯片支持哪种模式就配置哪种。Mode 0 和 Mode 3 最常见。

## 3. SPI 分频系数 — 实战经验 (Prescaler Selection)

### 3.1 STM32F103 SPI 时钟分频

STM32F103 的 SPI 时钟来源于 APB 总线时钟，可选分频系数为：**2、4、8、16、32、64、128、256**。

以 72MHz 主频为例：

| 分频系数 | SPI1 (APB2 = 72MHz) | SPI2 (APB1 = 36MHz) |
| :--- | :--- | :--- |
| 2 | 36 MHz | 18 MHz |
| 4 | 18 MHz | 9 MHz |
| 8 | 9 MHz | 4.5 MHz |
| 16 | 4.5 MHz | 2.25 MHz |
| 32 | 2.25 MHz | 1.125 MHz |
| 64 | 1.125 MHz | 562.5 kHz |
| **128** | **562.5 kHz** | **281.25 kHz** |
| 256 | 281.25 kHz | 140.625 kHz |

> **注意**：SPI1 挂在 APB2 (最高 72MHz)，SPI2 挂在 APB1 (最高 36MHz)，同样的分频系数下 SPI1 速率是 SPI2 的两倍。

### 3.2 实际项目案例 — 分频系数导致 SCLK 波形失真

**问题描述**：

在实际项目中使用 SPI 进行 OTA 升级时，初始配置分频系数为 16（SPI1 对应 4.5MHz），通过示波器观察发现 SCLK 波形严重失真（边沿变缓、振铃、过冲），导致通信数据错误。最终将分频系数调整为 128（562.5kHz）后，波形恢复正常，OTA 升级稳定通过。

```
分频 16 (4.5MHz) — SCLK 波形失真:

理想波形:    ┌──┐  ┌──┐  ┌──┐
          ───┘  └──┘  └──┘  └───

实际波形:    ╱╲    ╱╲    ╱╲        ← 上升沿变缓
          ──╱  ╲──╱  ╲──╱  ╲───   ← 振铃、过冲
                                    ← 从机采样时刻电平不确定


分频 128 (562.5kHz) — SCLK 波形正常:

实际波形:    ┌──┐  ┌──┐  ┌──┐     ← 边沿陡峭
          ───┘  └──┘  └──┘  └───  ← 无振铃
                                    ← 从机采样可靠
```

**失真原因分析**：

| 原因 | 说明 |
| :--- | :--- |
| **PCB 走线寄生参数** | 走线过长或未做阻抗匹配，频率越高，寄生电容/电感效应越明显 |
| **从机芯片实际能力** | 部分芯片标称支持高速，但实际在特定条件下表现不佳 |
| **GPIO 驱动能力不匹配** | GPIO Speed 配置（2MHz/10MHz/50MHz）必须大于 SCLK 频率 |
| **OTA 场景特殊性** | 大量连续数据传输，累积错误率放大，偶发错误变成必现故障 |

### 3.3 分频系数选择策略

> **SPI 速率不是越快越好，要根据实际硬件环境选择合适的分频系数。**

推荐调试流程：

```
步骤1: 大分频 (低速) 跑通功能
  │     确认协议层、数据收发没有逻辑问题
  ▼
步骤2: 逐步减小分频 (提速)
  │     每调一级，用示波器观察 SCLK 和数据线波形
  ▼
步骤3: 找到波形劣化临界点
  │     观察上升沿/下降沿时间，是否有振铃和过冲
  ▼
步骤4: 退一档作为最终配置
        留有裕量，确保在温度、电压波动下仍然稳定
```

> **OTA 场景补充**：Flash 写入本身有等待时间（典型 tPP = 几百 μs ~ 几 ms），SPI 速率往往不是 OTA 的瓶颈。稳定性远比速度重要。

## 4. SPI vs I2C vs RS485 横向对比 (Communication Comparison)

### 4.1 基本特性对比

| 对比项 | SPI | I2C | RS485 (Modbus) |
| :--- | :--- | :--- | :--- |
| **线数** | 4 根 (MOSI/MISO/SCLK/CS) | 2 根 (SDA/SCL) | 2 根 (A/B 差分) |
| **双工模式** | **全双工** | 半双工 | 半双工 |
| **寻址方式** | 硬件片选 (CS 拉低) | 软件地址 (7/10bit) | 协议地址 (Modbus 站号) |
| **速度** | 最快 (几十 MHz 级别) | 较慢 (100k/400k/1MHz) | 取决于波特率 (通常 9600~115200) |
| **距离** | 短距 (板级，几十 cm) | 短距 (板级 ~ 几米) | 长距 (最远 1200m) |
| **从机数量** | 受 CS 引脚数限制 | 受地址数限制 (最多 127) | 受协议地址限制 (最多 247) |
| **典型应用** | Flash、LCD、ADC/DAC | 传感器、EEPROM、RTC | 工业设备间通讯 |

### 4.2 寻址方式对比 — 为什么 SPI 速度最快

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         三种通讯的寻址开销对比                                 │
└─────────────────────────────────────────────────────────────────────────────┘

SPI — 硬件片选，零开销:

    CS:   ──┐                                    ┌──
            └────────────────────────────────────┘
    SCLK:    ┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐┌┐
          ───┘└┘└┘└┘└┘└┘└┘└┘└┘└┘└┘└┘└┘└┘└┘└┘└──
    MOSI:  ──[      直接开始传数据，无地址开销      ]──
             ↑
          CS 拉低 = 选中，立刻传数据


I2C — 软件地址，每次通讯都有开销:

    SDA:  ──┐                                                          ┌──
            └──[START]──[7bit地址][R/W]──[ACK]──[数据]──[ACK]──[STOP]──┘
            ↑                            ↑              ↑
         起始条件                    从机应答(耗时)  每字节都要ACK


RS485 (Modbus) — 协议帧地址，开销最大:

    数据:  [从机地址][功能码][数据区...][CRC16]  →  等待从机响应  →  [响应帧]
           ↑                  ↑                    ↑
        1字节地址          CRC计算耗时          响应等待时间 (ms级)
```

**SPI 比 I2C 快的四个原因**：

| 原因 | SPI | I2C |
| :--- | :--- | :--- |
| **寻址开销** | CS 拉低即选中，零开销 | 每次通讯需发 START + 7bit 地址 + R/W + 等待 ACK |
| **双工模式** | 全双工，MOSI/MISO 同时收发 | 半双工，SDA 线同一时刻只能单向 |
| **应答机制** | 无 ACK，数据连续传输 | 每传 1 字节等待 1 个 ACK |
| **时钟上限** | 仅受硬件能力限制 (STM32F103 最高 18MHz) | 协议限制 (标准 100kHz，快速 400kHz) |

### 4.3 可靠性对比 — SPI 的最大短板

SPI 速度快的代价是 **缺乏可靠性机制**。I2C 有 ACK 应答，Modbus 有 CRC16 校验，而 SPI 本身是"盲发"协议。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         可靠性机制对比                                        │
└─────────────────────────────────────────────────────────────────────────────┘

SPI — 无应答，盲发:

    主机: ──[发送数据]──[发送数据]──[发送数据]──
              ↓            ↓            ↓
    从机:  收到了？不知道  收到了？不知道  收到了？不知道
    
    主机完全不知道从机是否正确接收，数据可能静默损坏。


I2C — 逐字节 ACK 应答:

    主机: ──[发送数据]──等待──[发送数据]──等待──[发送数据]──等待──
              ↓           ↑       ↓          ↑       ↓          ↑
    从机:  [收到] ──[ACK]─┘    [收到] ──[ACK]─┘    [忙!] ──[NACK]─┘
                                                              ↑
                                                    主机立刻感知到问题！


RS485 (Modbus) — 帧级 CRC 校验:

    主机: ──[帧头][地址][功能码][数据][CRC16]──等待响应──
              ↓                                    ↑
    从机:  [校验CRC] → CRC 正确 → [响应帧+CRC] ──┘
                     → CRC 错误 → 不响应 → 主机超时重发
```

**可靠性排序**：

| 维度 | SPI | I2C | RS485 (Modbus) |
| :--- | :--- | :--- | :--- |
| **协议层校验** | 无 | ACK 应答 | CRC16 校验 |
| **错误感知能力** | 弱 (盲发) | 中 (字节级应答) | 强 (帧级校验) |
| **需要上层补偿** | 多 | 少 | 最少 |

### 4.4 SPI 可靠性补偿方案

既然 SPI 协议层不提供可靠性保障，需要在应用层自行实现：

| 补偿方案 | 说明 | 适用场景 |
| :--- | :--- | :--- |
| **回读校验** | 写入后读出比对 | Flash 写入、寄存器配置 |
| **状态寄存器轮询** | 操作前读状态确认芯片 ready | Flash 编程/擦除 |
| **CRC 校验** | 数据包中加入 CRC | SD 卡 SPI 模式自带；自定义协议 |
| **超时机制** | 等待响应时设超时 | 防止死等 |

### 4.5 综合选型建议

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         三种通讯方式选型指南                                   │
└─────────────────────────────────────────────────────────────────────────────┘

                    速度优先              引脚节省             长距离/工业
                       │                    │                    │
                       ▼                    ▼                    ▼
                    ┌─────┐             ┌─────┐             ┌──────┐
                    │ SPI │             │ I2C │             │ RS485│
                    └─────┘             └─────┘             └──────┘
                       │                    │                    │
               板级高速传输          板级多设备低速         远距离抗干扰
               Flash/LCD/ADC       传感器/EEPROM/RTC     工业设备间通讯
                       │                    │                    │
                  全双工/无校验        半双工/ACK          半双工/CRC16
                  需上层补偿可靠性     协议自带基本保障      协议保障最完善
```

> **核心原则**：SPI 把时间花在传数据上 (快)，I2C 把时间花在握手确认上 (稳)，Modbus 把时间花在帧封装和校验上 (最稳但最慢)。

## 5. SPI 引脚消耗问题与解决方案

SPI 每增加一个从机就需要多一根 CS 线，这是 SPI 的主要缺点之一。

### 5.1 多从机的引脚消耗

```
直连方式 — 每个从机独占一根 CS:

    MCU
    ┌──────┐
    │ CS0  │────► 从机1 (Flash)
    │ CS1  │────► 从机2 (LCD)
    │ CS2  │────► 从机3 (ADC)
    │ CS3  │────► 从机4 (DAC)
    │ MOSI │────► 共享
    │ MISO │◄─── 共享
    │ SCLK │────► 共享
    └──────┘

4 个从机 = 4 根 CS + 3 根共享 = 7 根线
8 个从机 = 8 根 CS + 3 根共享 = 11 根线
```

### 5.2 缓解方案：译码器扩展

```
译码器方式 — 3 根线控制 8 个从机:

    MCU                  74HC138
    ┌──────┐           ┌─────────┐
    │ PA0  │──────────►│ A    Y0 │────► CS0 (从机1)
    │ PA1  │──────────►│ B    Y1 │────► CS1 (从机2)
    │ PA2  │──────────►│ C    Y2 │────► CS2 (从机3)
    │      │           │      Y3 │────► CS3 (从机4)
    │ MOSI │────►      │      Y4 │────► CS4 (从机5)
    │ MISO │◄───       │      Y5 │────► CS5 (从机6)
    │ SCLK │────►      │      Y6 │────► CS6 (从机7)
    └──────┘           │      Y7 │────► CS7 (从机8)
                       └─────────┘

3 根地址线 + 3 根共享 = 6 根线，控制 8 个从机。
```

## 6. 常见问题与调试 (Troubleshooting)

### Q1: SCLK 波形失真 (边沿变缓、振铃)

**可能原因**：
1. 分频系数太小，SPI 速率超出硬件承受能力
2. PCB 走线过长或未做阻抗匹配
3. GPIO Speed 配置低于 SCLK 频率

**解决方案**：
1. 增大分频系数，从低速开始逐步提速
2. 缩短 SPI 走线，保持走线等长
3. GPIO Speed 至少设为 `GPIO_SPEED_FREQ_HIGH` (50MHz)

### Q2: OTA 升级数据校验失败

**可能原因**：
1. SPI 速率过高导致 bit 翻转，但 SPI 无应答机制，数据静默损坏
2. Flash 写入后未等待编程完成 (tPP)

**解决方案**：
1. 降低分频系数，确保波形质量
2. 写入后轮询 Flash 状态寄存器，确认 BUSY 标志清除
3. 写入后回读校验

### Q3: 从机无响应 (MISO 始终为高阻)

**可能原因**：
1. CS 引脚未正确拉低
2. SPI 模式 (CPOL/CPHA) 与从机不匹配
3. 从机芯片未上电或损坏

**解决方案**：
1. 用示波器确认 CS 电平
2. 查阅从机数据手册确认 SPI 模式
3. 检查从机供电

### Q4: 通信时好时坏

**可能原因**：
1. 分频系数处于临界值，受温度/电压波动影响
2. SPI 信号线旁有高频干扰源 (如天线)
3. 多任务环境下 SPI 总线未做互斥保护

**解决方案**：
1. 分频系数退一档使用，留有裕量
2. SPI 走线远离天线和高频信号
3. 使用互斥信号量保护 SPI 总线访问
